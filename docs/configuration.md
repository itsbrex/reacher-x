# ReacherX Configuration

This guide explains where ReacherX environment variables belong, which features need them, and which operational controls can be tuned without changing application code.

Start from [`.env.example`](../.env.example). Never commit `.env.local`, API keys, webhook secrets, encryption keys, or deployment credentials.

## Environment Locations

ReacherX has two separate environment-variable stores:

| Location                                   | Used by                                                                          | Examples                                                                            |
| ------------------------------------------ | -------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| `.env.local` and hosting-provider settings | Next.js browser/server runtime and Convex CLI deployment selection               | `NEXT_PUBLIC_CONVEX_URL`, WorkOS AuthKit values, `NEXT_PUBLIC_SITE_URL`, PostHog    |
| Convex deployment environment              | Convex queries, mutations, actions, workflows, agents, and provider integrations | AI/provider keys, model roles, prospecting controls, rate limits, workpool controls |

Convex functions do **not** load backend values from `.env.local`. Variables must be configured separately for each Convex deployment. See the official [Convex environment-variable guide](https://docs.convex.dev/production/environment-variables) and [`npx convex env` reference](https://docs.convex.dev/cli/reference/env).

Do not pass the complete `.env.local` or `.env.example` file to `npx convex env set --from-file`. Those files intentionally mix frontend, local-only, public, and backend variables.

## Initial Local Setup

Create the local Next.js environment file:

```bash
cp .env.example .env.local
```

Configure or select a Convex development deployment and validate the backend:

```bash
npx convex dev --once
```

The command writes `CONVEX_DEPLOYMENT` and `NEXT_PUBLIC_CONVEX_URL` to `.env.local`. If the first push reports a missing provider variable, the deployment has still been selected; set the missing value and run the command again.

Set sensitive Convex values interactively so they do not appear in shell history:

```bash
npx convex env set WORKOS_CLIENT_ID
npx convex env set OPENROUTER_API_KEY
```

By default, `npx convex env` targets the selected development deployment. Use `--prod` only when intentionally changing production:

```bash
npx convex env set --prod VARIABLE_NAME
```

Use the Convex dashboard to confirm variable presence without printing values. The installed CLI can also list variables, but its output includes secret values, so run it only in a private terminal:

```bash
npx convex env list
npx convex env list --prod
```

After setup, keep the backend and frontend running in separate terminals:

```bash
# Terminal 1
npx convex dev
```

```bash
# Terminal 2
pnpm dev
```

## Minimum Configuration By Feature

### Authentication And Core AI

Configure these for an authenticated, AI-enabled application:

| Variable                          | Location     | Purpose                                                              |
| --------------------------------- | ------------ | -------------------------------------------------------------------- |
| `NEXT_PUBLIC_CONVEX_URL`          | `.env.local` | Convex client URL; normally generated by `npx convex dev`            |
| `WORKOS_CLIENT_ID`                | Both         | AuthKit client ID and Convex JWT audience/JWKS configuration         |
| `WORKOS_API_KEY`                  | `.env.local` | AuthKit server credential                                            |
| `WORKOS_COOKIE_PASSWORD`          | `.env.local` | AuthKit session-cookie encryption; must be at least 32 characters    |
| `NEXT_PUBLIC_WORKOS_REDIRECT_URI` | `.env.local` | AuthKit callback; locally `http://localhost:3000/callback`           |
| `NEXT_PUBLIC_SITE_URL`            | Both         | Public web origin used when constructing auth and provider callbacks |
| `OPENROUTER_API_KEY`              | Convex       | Primary model provider for agents and AI workflows                   |

The WorkOS redirect URI must also be allowed in the WorkOS dashboard. See the official [AuthKit Next.js SDK guide](https://workos.com/docs/sdks/authkit-nextjs).

## AI Model Routing

Model variable names represent stable workloads, not permanent model families. Change their values when models evolve; application code should continue referring to the workload role.

Values are OpenRouter model IDs unless otherwise noted.

| Variable                     | Workload                                                  |
| ---------------------------- | --------------------------------------------------------- |
| `AI_FAST_MODEL`              | Fast general generation path                              |
| `AI_REASONING_MODEL`         | Higher-judgment general reasoning path                    |
| `AI_AUTOCOMPLETE_MODEL`      | Autocomplete and lightweight helper generation            |
| `AI_SETUP_AGENT_MODEL`       | Workspace setup agent and lower-cost tool-calling work    |
| `AI_MAIN_AGENT_MODEL`        | Main prospect conversation and outreach agent             |
| `AI_OUTREACH_ROUTER_MODEL`   | Routes outreach work to the appropriate generation lane   |
| `AI_OUTREACH_FAST_MODEL`     | Fast outreach generation lane                             |
| `AI_OUTREACH_STANDARD_MODEL` | Standard outreach generation lane                         |
| `AI_OUTREACH_RECOVERY_MODEL` | Recovery lane for difficult or failed outreach generation |
| `AI_VISION_MODEL`            | Image/GIF understanding for multimodal turns              |
| `AI_TEXT_EMBEDDING_MODEL`    | Agent-memory and RAG embeddings                           |

`OPENROUTER_ROUTING_PRESET` is the fallback for `AI_FAST_MODEL` and `AI_REASONING_MODEL` when those role variables are unset:

- `current`: preserves the current fast/reasoning provider strategy.
- `cost_optimized`: uses the cost-optimized fast/reasoning strategy and is the code default.

Role-specific model variables take precedence over the preset. A newly selected model uses generic OpenRouter provider fallback unless the code has a model-specific optimized route.

### Embedding Compatibility

The RAG component is configured for exactly 1536 dimensions. Any value assigned to `AI_TEXT_EMBEDDING_MODEL` must return 1536-dimensional vectors. Changing to a model with a different dimension requires a coordinated code/index migration and re-embedding existing content.

## Real Prospecting Scheduling

These settings affect real workflows only; they do not control onboarding previews.

| Variable                                                    |                     Code fallback | Meaning                                                                                    |
| ----------------------------------------------------------- | --------------------------------: | ------------------------------------------------------------------------------------------ |
| `PROSPECTING_AUTO_RESCHEDULE`                               | Production: `1`; development: `0` | Enables continuous scheduling. Set explicitly to `0` in development and `1` in production. |
| `PROSPECTING_RESCHEDULE_INTERVAL_MINUTES`                   |                              `60` | Normal interval after bootstrap finishes                                                   |
| `PROSPECTING_BOOTSTRAP_READY_TARGET`                        |                              `12` | Ready-prospect count that ends accelerated scheduling                                      |
| `PROSPECTING_BOOTSTRAP_INTERVAL_MINUTES`                    |                               `2` | Delay between bootstrap scheduling decisions                                               |
| `PROSPECTING_BOOTSTRAP_MAX_CYCLES`                          |                               `8` | Maximum accelerated discovery cycles                                                       |
| `PROSPECTING_BOOTSTRAP_NO_PROGRESS_TIMEOUT_MINUTES`         |                              `20` | Ends acceleration only after this much time without meaningful progress                    |
| `PROSPECTING_BOOTSTRAP_PENDING_QUALIFICATION_YIELD_PERCENT` |                              `30` | Estimated percentage of pending qualification work used in the projected-ready count       |

The 20-minute timeout is **not** a total bootstrap-duration limit. Its timer resets when a cycle completes or ready, qualification, enrichment, or in-flight counts make meaningful progress. Provider waits are also bounded by the remaining no-progress window.

Before another discovery cycle starts, the scheduler projects whether ready prospects plus viable qualification/enrichment work can reach the target. If so, it waits for in-flight work instead of spending more provider capacity.

## SocialAPI And LinkdAPI Budgets

Budgets are shared at the provider level across users and workspaces. This prevents aggregate platform traffic from exceeding the configured account limits when multiple workspaces run concurrently.

| Variable                                    | Code fallback | Meaning                                                         |
| ------------------------------------------- | ------------: | --------------------------------------------------------------- |
| `SOCIALAPI_REQUESTS_PER_MINUTE`             |         `500` | Hard SocialAPI request budget                                   |
| `SOCIALAPI_TARGET_REQUESTS_PER_MINUTE`      |         `300` | Paced SocialAPI operating target; cannot exceed the hard budget |
| `SOCIALAPI_BUDGET_RESERVATION_MAX_ATTEMPTS` |          `12` | Attempts to reserve provider capacity                           |
| `SOCIALAPI_OCC_RETRY_BASE_MS`               |          `25` | Base retry delay after budget-record contention                 |
| `SOCIALAPI_OCC_RETRY_JITTER_MS`             |          `40` | Contention retry jitter                                         |
| `LINKDAPI_REQUESTS_PER_MINUTE`              |          `30` | Hard LinkdAPI request budget                                    |
| `LINKDAPI_TARGET_REQUESTS_PER_MINUTE`       |          `24` | Paced LinkdAPI operating target; cannot exceed the hard budget  |
| `LINKDAPI_BUDGET_RESERVATION_MAX_ATTEMPTS`  |          `12` | Attempts to reserve provider capacity                           |
| `LINKDAPI_OCC_RETRY_BASE_MS`                |          `25` | Base retry delay after budget-record contention                 |
| `LINKDAPI_OCC_RETRY_JITTER_MS`              |          `40` | Contention retry jitter                                         |

Set hard limits to the real provider-account allowance. Set target limits at or below the hard limits; lower targets leave headroom for latency, retries, provider enforcement differences, and traffic bursts.

## Deployment Checklist

Before enabling a feature in development or production:

1. Set its variables on the intended Convex deployment, not only in `.env.local`.
2. Confirm the selected deployment in the Convex dashboard. If you use `npx convex env list`, remember that it prints secret values.
3. Keep `PROSPECTING_AUTO_RESCHEDULE=0` in development unless continuous provider usage is intentional.
4. Confirm provider hard limits match the actual API account plan.
5. Configure provider callback and webhook URLs for the correct Convex site URL.
6. Test with development credentials before applying production values.
7. Never copy production secrets into development or commit them to the repository.
